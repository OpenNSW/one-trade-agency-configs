# Manifest

## Overview

Each agency in this repo has its own `manifest.json` at `agency-configs/<agency_name>/manifest.json`. The manifest is the machine-readable entry point for downstream consumers — it gives them a stable `taskCode → path` index without requiring them to walk the directory tree.

Manifests are generated by `scripts/generate-manifest.mjs` and committed to the repository. CI enforces that they are always in sync with the task configs on disk.

See [architecture.md](architecture.md) for a full description of the `agency-configs/` directory structure.

## Why per-agency, not one repo-level manifest?

`taskCode` uniqueness is scoped to an agency. The same `taskCode` value may exist in two different agencies. A single repo-level manifest would require a composite key (e.g. `agency/taskCode`), which is less ergonomic. Per-agency manifests keep the lookup simple: consumers fetch `agency-configs/<agency>/manifest.json` and look up by `taskCode` directly.

## Schema

```jsonc
{
  "version": "0.0.0-dev",               // tracks repo maturity; will become semver post-1.0
  "generated": "scripts/generate-manifest.mjs",
  "agency": "fcau",                          // the agency this manifest describes
  "tasks": {
    "<taskCode>": "<repo-root-path>",        // taskCode → path from repo root (under agency-configs/)
    ...
  }
}
```

### `tasks` map

Keys are `taskCode` values (sorted alphabetically for stable diffs). Values are paths relative to the repository root — the same base you use to construct a GitHub raw URL.

Example:

```json
{
  "tasks": {
    "fcau_application_review_v1": "agency-configs/fcau/task-configs/fcau_application_review_v1.json",
    "fcau_sample_assessment_v1":  "agency-configs/fcau/task-configs/fcau_sample_assessment_v1.json",
    "fcau_sample_decision_v1":    "agency-configs/fcau/task-configs/fcau_sample_decision_v1.json"
  }
}
```

## Consuming the manifest

The pattern is: fetch `<agency>/manifest.json`, look up `tasks[taskCode]` to get the path, then fetch that path.

```bash
BASE="https://raw.githubusercontent.com/OpenNSW/one-trade-agency-configs/main"
MANIFEST=$(curl -s "$BASE/agency-configs/fcau/manifest.json")
PATH=$(echo "$MANIFEST" | jq -r '.tasks["fcau_application_review_v1"]')
curl -s "$BASE/$PATH" | jq .
```

See [consumption.md](consumption.md) for full examples in bash, Node, Python, and Go.

## Keeping manifests in sync

Manifests must be regenerated and committed whenever task configs change.

```bash
npm run manifest   # regenerate all agency manifests
git add agency-configs/fcau/manifest.json   # (and any other changed manifests)
git commit -m "..."
```

`npm run check` enforces this: it regenerates manifests and then runs `scripts/check-manifests.mjs`, which uses `git status --porcelain` to detect both modified tracked manifests and new untracked manifests. CI will fail if a manifest is missing or stale.

## Version field

The `version` field tracks the maturity of this repository. It is currently `"0.0.0-dev"` for all manifests. Post-1.0, it will be bumped following semver:

- **Major**: breaking changes to the task config schema
- **Minor**: new agencies or new task codes
- **Patch**: content edits to existing task configs
