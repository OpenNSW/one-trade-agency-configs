# Architecture

## What this repository is

`one-trade-agency-configs` is a **language-agnostic JSON data repository**. It stores task configuration files for OneTrade agency integrations — pure data, no application code, no runtime.

Any language that can read JSON — Node, Python, Go, browser, mobile — consumes task configs identically. There are no package-manager dependencies for downstream consumers.

## What this repository is not

This repo does **not** contain form schemas or workflow templates. Those live in [`one-trade-templates`](https://github.com/OpenNSW/one-trade-templates), which is the single source of truth for all form and workflow definitions on the OneTrade platform.

## Relationship to `one-trade-templates`

| Concern | Repository |
|---------|-----------|
| Form schemas (JSON Schema + UI schema) | `one-trade-templates` |
| Workflow definitions | `one-trade-templates` |
| Task configurations (taskCode, meta, form references) | `one-trade-agency-configs` (this repo) |

Task configs reference form templates by ID. The `forms.view` and `forms.review` values in a task config are template IDs that resolve via the `one-trade-templates` manifest:

```
one-trade-agency-configs                   one-trade-templates
────────────────────────                   ───────────────────
agency-configs/fcau/task-configs/          manifest.json#byId
  fcau_application_review_v1.json
    forms.review = "fcau-apply-health-cert--reviewer-form"
                        └──────────────────────────────────► templates/fcau/1-application/reviewerinput_jsonform.json
```

## Directory layout

```
agency-configs/
└── <agency_name>/
    ├── manifest.json                  ← generated index for this agency
    └── task-configs/
        └── <task_code>.json           ← one file per task
```

All agencies live under `agency-configs/`. Tooling directories (`scripts/`, `docs/`, `.github/`) are at the repo root and are never treated as agencies.

## Per-agency manifest

Each agency has its own `manifest.json` (generated by `scripts/generate-manifest.mjs`). Unlike `one-trade-templates`, which has a single repo-level manifest, task configs are scoped per agency — `taskCode` uniqueness is guaranteed only within an agency, not across the whole repo. Per-agency manifests reflect this boundary.

See [manifest.md](manifest.md) for the full schema and consumer usage.

## Tooling

| Script | Purpose |
|--------|---------|
| `scripts/validate-configs.mjs` | Enforce structural rules on every task config under `agency-configs/` |
| `scripts/generate-manifest.mjs` | Write `agency-configs/<agency>/manifest.json` for every discovered agency |
| `scripts/check-manifests.mjs` | Verify all manifests are committed and in sync (used in CI) |

The `npm run check` command runs all three plus Prettier formatting checks. See [validation.md](validation.md) for the rules enforced.
